Análise de Processo - AgileTech Solutions

a) Aplicação do Manifesto Ágil

1. Indivíduos e interações > processos e ferramentas
Como a equipe é pequena, a comunicação direta é essencial. Priorizar interação ajuda a lidar com mudanças rápidas.

2. Software funcionando > documentação extensa
A empresa já teve problemas com documentação desatualizada. Logo, focar em entregas frequentes de software funcional é mais eficiente.

3. Colaboração com o cliente > negociação de contratos
O cliente é participativo (mesmo com pouco tempo), então é importante envolvê-lo continuamente no processo.

4. Responder a mudanças > seguir um plano
Os requisitos são vagos e mudam com frequência, então flexibilidade é essencial.

---

Por que usar Ágil ao invés de Cascata?

O modelo cascata não é adequado pois exige requisitos bem definidos desde o início. Como o projeto possui mudanças constantes e necessidade de entregas rápidas, métodos ágeis permitem adaptação contínua e entrega de valor incremental.

---

 Práticas ágeis recomendadas

- Scrum (organização do trabalho em sprints)
- Integração contínua
- Entrega contínua
- Pair Programming
- Feedback frequente com o cliente

---

b) Programação em Pares

Conceito
Pair programming é quando dois desenvolvedores trabalham juntos no mesmo código: um escreve (driver) e outro revisa (navigator).

Benefícios
- Menos bugs
- Compartilhamento de conhecimento
- Código mais limpo

Desafios no EAD
- Diferença de horários
- Comunicação limitada
- Dependência de ferramentas online

Adaptações
- Uso de ferramentas como VS Code Live Share
- Revisões assíncronas (code review como alternativa)

c) Dificuldades de Brooks

Mais relevantes:
- Complexidade → sistema de gestão de projetos
- Mutabilidade → requisitos mudam sempre
- Invisibilidade → difícil visualizar software

Mitigação com ágil:
- Sprints curtas reduzem complexidade
- Feedback constante lida com mudanças
- Kanban/Scrum aumentam visibilidade do progresso
